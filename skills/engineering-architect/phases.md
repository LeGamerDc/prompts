# 阶段详细指导

各阶段的详细指导、询问模板、质量检查、挑战时机。配合 [SKILL.md](SKILL.md) 中的阶段概览和对话流程控制使用。

---

## 渐进式披露模型

所有文档和索引设计的核心框架：

```
L0 (始终加载, Cursor Rule 自动注入, ≤200 行)
├── .cursor/rules/project.mdc   # 项目全局规则 (6 板块)
├── docs/status.md              # 当前进度与待办

L1 (按意图加载, L0 中有导航索引)
├── docs/design/           # 需求/技术方案 (RA/TA 产出)
├── docs/architecture/     # 架构文档 (按模块拆分)
├── docs/guides/           # 开发指南/规范

L2 (具体任务时加载)
├── docs/specs/            # 接口/协议 spec
├── docs/modules/<mod>/    # 模块级详细文档
├── <代码目录>/README.md   # 代码级文档
```

| 层级 | 加载时机 | 内容特征 | 大小约束 |
|------|---------|---------|---------|
| L0 | Agent 启动时自动注入 | 全局上下文、索引、核心规则 | ≤200 行 / ~1500 tokens |
| L1 | 根据任务意图按需读取 | 模块级架构、技术方案、开发指南 | 每文件 ≤400 行 |
| L2 | 执行具体开发时读取 | 详细 spec、代码结构、API 文档 | 每文件 ≤300 行 |

超出大小约束时，拆分为子文档并在原文档中保留摘要和引用链接。

## 阶段 0：输入导入与模式判断

**判断逻辑**：
1. 检查 `docs/plan/Technical.md` 是否存在
   - **存在** → 模式 A，读取 RA/TA 产出
   - **不存在** → 询问用户：新项目还是现有项目？

2. 模式 A 动作：
   - 读取 `Requirement.md`、`Technical.md`、`DecisionRecords.md`
   - 复述理解：项目目标、技术选型、组件拆分、任务规划
   - 与用户确认

3. 模式 B 动作（现有项目）：
   - 使用 subagent 并行扫描：代码结构、文档目录、工具配置、协议规范
   - 向用户汇报扫描结果并确认

**质量检查**：
- [ ] 项目的技术栈是否明确
- [ ] 项目的模块/组件划分是否清晰
- [ ] 现有项目：是否识别了主要代码模块、工具、配置、协议

## 阶段 1：工程目录设计

### 1.1 代码目录设计

**必须明确的要素**：
1. **顶层结构** - 根目录下有哪些一级目录？
2. **模块划分** - 代码如何按模块/组件组织？
3. **公共层** - 共享代码（工具库/类型定义/常量）放在哪里？
4. **边界规则** - 模块间的引用规则？

**设计原则**：
- 每个模块/组件有独立目录，便于独立开发
- 目录层级不超过 4 层
- 模块内有明确入口文件（index/main/mod）
- 公共代码集中管理

**询问模板**：
- 基于技术方案中的组件拆分，你倾向于怎样的目录组织方式？
- 共享代码放在哪里？
- 对目录层级深度有什么偏好？

### 1.2 文档目录设计

根据渐进式披露模型设计 `docs/` 结构。

**讨论点**：
- 文档是集中在 `docs/` 还是就近放在代码目录中？
- 现有 RA/TA 产出文档路径是否需要调整？
- 哪些模块需要 L2 级别的详细文档？

### 1.3 现有项目特殊处理

1. **差异分析**：对比现有结构与推荐结构
2. **迁移讨论**：哪些差异需要修复、哪些可接受
3. **优先级排序**：P0 建索引 → P1 补文档 → P2 调结构

**质量检查**：
- [ ] 每个模块/组件是否有独立目录
- [ ] 目录层级是否不超过 4 层
- [ ] 文档目录是否覆盖 L0/L1/L2 三层
- [ ] 模块边界与目录边界是否对齐

**挑战时机**：
- 结构过于扁平（文件混在一起）→ 建议按模块组织
- 结构过深 → 建议扁平化，减少 Agent 探索成本
- 文档目录与代码目录不一致 → 讨论对齐策略

## 阶段 2：文档工程构建

**2.1 状态文档** — 创建 `docs/status.md`，记录项目状态、模块进度、近期变更、待办。

**2.2 模块文档** — 为每个模块创建 L1 入口（`docs/architecture/<module>.md`），包含：模块职责、技术选型、对外接口、依赖关系、关键设计决策。

**2.3 开发指南** — 根据项目需要创建 `docs/guides/` 下的文档（code-style / error-handling / commands 等）。

**2.4 现有项目资源索引** — 扫描并记录：
- **代码索引**：关键模块入口、公共工具库、类型定义位置
- **工具索引**：Makefile targets、Scripts 功能
- **配置索引**：环境配置、构建配置、第三方服务配置
- **协议索引**：Proto 文件、API 定义、数据 Schema

索引信息写入 L1/L2 文档，并在 L0 Rule 文档导航中建立索引入口。

**质量检查**：
- [ ] 每个模块是否有文档入口
- [ ] 文档间的引用链接是否正确
- [ ] 现有项目：关键资源是否全部被索引
- [ ] 文档是否符合大小约束

## 阶段 3：Spec 与进度文档（可选）

**触发**：询问用户是否需要。如果 TA 产出中有明确的组件拆分和任务 DAG，建议生成。

**3.1 Spec 文档** — 为关键接口生成 spec 模板（模块间接口/外部 API/协议），放入 `docs/specs/`（L2 层）。

**3.2 进度追踪** — 基于 TA 阶段规划生成进度追踪文档，与 `docs/status.md` 关联。

**询问模板**：
- 技术方案中定义了 [N] 个组件和 [M] 个任务。需要生成 spec 文档模板吗？
- 需要进度追踪文档吗？倾向于什么粒度？

---

## 阶段 4：代码脚手架（可选）

**触发**：询问用户。新项目建议生成，现有项目通常跳过。

**可生成的内容**：
1. 构建配置 — Makefile / package.json / go.mod 等
2. 模块骨架 — 每个模块的目录 + 入口文件 + 导出声明
3. 接口桩 — 模块间接口的类型定义
4. 基础设施 — 日志、配置加载等通用工具桩代码

**质量检查**：
- [ ] 脚手架是否可正常编译/运行（空壳状态）
- [ ] 模块结构是否与阶段 1 的目录设计一致
- [ ] 接口桩是否与 TA 的组件拆分一致

## 阶段 5：Cursor Rules 制定

### 5.1 项目级 Rule (`project.mdc`)

Agent 入口文件，始终自动加载，≤200 行。6 个板块：

| 板块 | 内容要点 | 来源 |
|------|---------|------|
| 项目目标 | 一句话描述 + 阶段 + 核心约束（~3-5 行） | RA 概要 |
| 架构边界 | 架构风格 + 模块清单 + 依赖规则 + 目录速览（内联摘要 + L1 索引） | TA + 阶段 1 |
| 开发原则 | 文档先行/搜索优先/探索隔离等（~5-10 条，与用户讨论确定） | 讨论 |
| 工具命令 | 构建/测试/lint 等常用命令表格 | 扫描/TA |
| 代码风格 | 命名/错误处理/导入顺序（核心 3-5 条 + L1 索引） | TA + 约定 |
| 文档导航 | L0/L1/L2 分层索引地图 | 阶段 1-2 |

### 5.2 工作流 Rule (`workflow.mdc`)

4 类行为规范：
- **文档加载协议**：L0 自动加载 → L1 按意图 → L2 具体开发 → 禁止未读就编码
- **文档维护协议**：改代码更新文档 → 新模块更新导航 → 变更接口更新 spec → 记录到 status.md
- **探索协议**：subagent 探索 → 先查文档再 grep → 发现回写文档
- **开发范式**：搜索优先 → 单次聚焦 → 接口先行 → 最小变更

**讨论点**：是否需要额外规则？严格程度？是否按文件类型设置不同规则？可合并或拆分。

**质量检查**：
- [ ] project.mdc 是否覆盖 6 个板块且 ≤200 行
- [ ] workflow.mdc 是否覆盖 4 类行为规范
- [ ] 规则是否具体可执行
- [ ] 规则之间是否无矛盾

## 特殊场景处理

### 场景 1：无 RA/TA 输入的新项目
引导用户提供基本信息（项目目标/技术栈/主要模块），生成最小可用工程结构，建议后续补充 RA/TA。

### 场景 2：大型现有项目
分批次扫描，优先索引核心模块和公共工具。L0 只列关键模块，其余放 L1。迁移计划分期执行。

### 场景 3：Monorepo / 多子项目
讨论子项目间独立性和共享代码。每个子项目可有自己的 L1 文档。L0 列出所有子项目及关系。Rules 可按子项目拆分。

### 场景 4：已有部分文档
识别评估现有文档质量，复用合理的，补充缺失的。在 L0 索引中统一纳管。

### 场景 5：用户要求最小化
只生成 L0 Cursor Rule + 基础 workflow Rule，跳过 L1/L2 文档、spec、脚手架，后续按需补充。

### 场景 6：中途修改
接受修改请求，评估对后续阶段的影响，更新已生成文档和规则。影响范围大时回到受影响阶段重新讨论。

## 主动挑战触发点

每轮对话后主动检视：

1. **目录结构是否支持独立工作** — Agent 能否只看一个模块目录就理解该模块？模块间依赖是否通过明确接口体现？
2. **文档分层是否合理** — L0 是否超 token 预算或缺关键上下文？L1/L2 边界是否清晰？
3. **规则是否具体可执行** — 是否过于空泛或过于死板？
4. **索引是否实用** — 太粗（找不到）还是太细（维护成本高）？
5. **迁移策略是否渐进** — 是否一步到位（风险高）？是否保留回退路径？
