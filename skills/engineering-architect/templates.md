# 文档模板与质量自检

生成最终产出前，先执行质量自检清单，再按模板生成文档。

---

## 质量自检清单

### 结构检查
- [ ] 目录结构是否支持模块独立开发
- [ ] 目录层级是否不超过 4 层
- [ ] 每个模块是否有明确入口文件

### 文档检查
- [ ] L0（Cursor Rule）是否 ≤200 行 / ~1500 tokens
- [ ] L0 是否覆盖 6 个板块（项目目标/架构边界/开发原则/工具命令/代码风格/文档导航）
- [ ] L1 是否覆盖所有模块/组件
- [ ] 文档索引是否覆盖所有关键资源
- [ ] 文档间的引用链接是否有效
- [ ] 文档大小是否在约束范围内

### 规则检查
- [ ] Cursor Rule 是否覆盖 4 类行为规范（加载/维护/探索/开发）
- [ ] 规则是否具体可执行
- [ ] 规则之间是否无矛盾

### 现有项目检查（如适用）
- [ ] 代码索引是否覆盖关键模块和公共工具
- [ ] 工具索引是否覆盖 Makefile/scripts
- [ ] 配置索引是否覆盖关键配置文件
- [ ] 协议索引是否覆盖 proto/API 定义
- [ ] 索引是否不与现有结构冲突

如有检查项未通过，需要与用户进一步沟通补充。

## 工作流 Cursor Rule 模板 (`.cursor/rules/workflow.mdc`)

```markdown
---
description: Agent 工作流规范
globs:
alwaysApply: true
---

# Agent 工作流规范

## 文档加载协议

1. 任务开始时，`project.mdc` 已自动加载（L0）
2. 根据任务意图，从 L0 文档导航中找到相关 L1 文档并读取
3. 执行具体开发时，读取 L2 文档（spec、模块详情、代码 README）
4. **禁止在没有读取相关文档的情况下开始编码**

## 文档维护协议

1. 修改代码后，检查并更新对应的架构/模块文档
2. 新增模块/文件时，更新 L0 文档导航 + L1 模块文档
3. 变更接口时，更新 spec 文档（如有）
4. 每次有意义的变更，在 `docs/status.md` 中记录摘要

## 探索协议

1. 探索不熟悉的代码/知识时，使用 subagent 执行（减少主 context 污染）
2. 搜索代码前，先查阅 L0 文档导航定位目标区域
3. 探索中发现的重要信息，回写到对应文档

## 开发范式

1. **搜索优先**：实现新功能前，先搜索项目中是否已有类似代码或工具
2. **单次聚焦**：一次只处理一个模块/任务，通过文档获取跨模块信息
3. **接口先行**：先定义接口/类型，再实现具体逻辑
4. **最小变更**：只修改任务要求的部分，不做顺手重构
```
