# 阶段详细指导

各阶段的详细询问模板、质量检查清单、挑战时机。配合 [SKILL.md](SKILL.md) 中的阶段概览和对话流程控制使用。

---

## 阶段 0：需求导入与理解

**询问**：
- 请提供需求文档的路径或内容
- 需求文档中有没有你认为对技术方案影响最大的要素？

**完成动作**：
1. 读取需求文档
2. 用自己的语言复述核心内容：背景与问题、任务目标、任务范围、交付方案、风险与前提
3. 与用户确认理解是否准确

**质量检查**：
- [ ] 需求文档是否完整（包含背景与问题、任务目标、任务范围、交付方案、风险与前提）
- [ ] 对需求的理解是否与用户确认一致
- [ ] 是否识别出需求中隐含的技术约束

**挑战时机**：
- 需求文档不完整 → 指出缺失部分，建议先补充需求文档
- 需求范围过于宏大 → 建议先与需求方确认是否需要拆分

## 阶段 1：技术约束识别

**必须明确的要素**：
1. **现有技术栈** - 项目/团队已经在使用什么技术？
2. **基础设施约束** - 部署环境、服务器资源、网络条件等
3. **第三方依赖** - 需要对接的外部系统/服务/API
4. **团队能力边界** - 团队熟悉的技术、学习成本承受度
5. **非功能性要求** - 性能指标、安全要求、可用性要求等

**询问模板**：
- 当前项目使用的技术栈是什么？有没有必须沿用的技术？
- 部署环境是什么？（云服务/自建机房/边缘设备/其他）
- 这个需求需要对接哪些外部系统或服务？
- 团队最熟悉的技术是什么？有没有明确不想引入的技术？
- 有没有明确的性能/安全/可用性要求？

**质量检查**：
- [ ] 硬约束（不可变更的）是否已全部识别
- [ ] 软约束（可协商的）是否已标注
- [ ] 约束之间是否存在冲突

**挑战时机**：
- 约束过多导致方案空间极小 → 讨论哪些约束可以松动
- 用户声称"没有约束" → 引导识别隐含约束（预算、时间、人员等）

---

## 阶段 2：技术选型讨论

**必须明确的要素**：
1. **需要决策的技术点** - 梳理出所有需要做选型的关键技术点
2. **候选方案** - 每个技术点的可选方案
3. **评估维度** - 成熟度/社区支持/性能/学习成本/与现有栈的兼容性
4. **最终选择** - 每个技术点的决策结论

**每个关键决策的讨论流程**：
1. 明确决策点：我们需要为 [什么] 选择技术方案
2. 列出候选方案：考虑到约束，有 [A/B/C] 几种选择
3. 逐一评估：从 [评估维度] 分析各方案优劣
4. 做出决策：选择 [方案]，因为 [理由]
5. 记录到决策文档：完整推理过程写入决策记录

**询问模板**：
- 对于 [技术点]，你有没有倾向的方案？
- 在 [方案 A] 和 [方案 B] 之间，你更看重哪些因素？
- 这个选型有没有你之前的经验可以参考？

**质量检查**：
- [ ] 所有关键技术点是否都已讨论
- [ ] 每个选型是否在约束空间内
- [ ] 选型之间是否存在兼容性问题
- [ ] 决策理由是否充分

**挑战时机**：
- 用户选择了团队不熟悉的技术 → 讨论学习成本和风险
- 选型过于保守/激进 → 讨论是否有更平衡的选择
- 用户没有考虑替代方案 → 主动提出至少一个替代方案进行比较

## 阶段 3：流程与数据概要

### 3.1 工作流程

**必须明确**：核心流程、流程步骤、角色/系统交互（谁触发？谁参与处理？）

### 3.2 数据流程

**必须明确**：数据类型、数据流向、数据转换

### 3.3 数据概要

**必须明确**：需要存储的数据、生产方、消费方、生命周期（创建/更新/过期/删除策略）

**询问模板**：
- 用户/系统的主要操作流程是什么？请描述关键步骤
- 系统需要存储哪些数据？这些数据是怎么产生的？
- 这些数据最终会被谁使用？用在什么场景下？
- 有没有需要在多个组件之间共享的数据？

**质量检查**：
- [ ] 核心流程是否完整覆盖需求范围
- [ ] 数据的生产和消费关系是否清晰
- [ ] 是否存在数据孤岛（产生但无人消费，或需要但无人产生）

**挑战时机**：
- 流程过于复杂 → 讨论是否可以简化或分阶段实现
- 数据流向不清晰 → 要求用户举具体场景来说明
- 存在数据一致性风险 → 主动提出讨论

## 阶段 4：组件拆分

**必须明确的要素**：
1. **组件列表** - 系统拆分为哪些组件？
2. **组件职责** - 每个组件负责什么？（单一职责）
3. **组件边界** - 组件之间的界限在哪里？
4. **通信方式** - 组件之间如何交互？（同步/异步/事件驱动/共享数据）
5. **解耦策略** - 如何确保组件之间低耦合？

**询问模板**：
- 基于前面讨论的流程，你觉得系统可以拆分为哪些主要模块？
- 每个模块的核心职责是什么？
- 模块之间需要怎样的交互？有没有特别紧密耦合的部分？
- 有没有可以独立开发、独立测试的模块？

**质量检查**：
- [ ] 每个组件是否有明确的单一职责
- [ ] 组件之间的依赖关系是否清晰
- [ ] 是否存在循环依赖
- [ ] 组件划分是否支持并行开发

**挑战时机**：
- 某个组件职责过多 → 建议进一步拆分
- 组件之间耦合度过高 → 讨论解耦方案
- 拆分粒度过细 → 讨论是否过度设计

---

## 阶段 5：任务拆分（DAG）

**任务拆分原则**：
1. **逻辑解耦** - 每个任务应有明确的输入、输出和验收条件
2. **粒度适中** - 单个任务的 context 应可控，避免过大或过碎
3. **同组件可拆多任务** - 同一个组件可以拆分为"基本功能"和"补充功能"等多个任务，分布在不同阶段
4. **依赖显式化** - 任务之间的前置依赖必须标注清楚

**必须明确的要素**：任务清单、任务描述、输入/输出、验收条件、依赖关系、工作量估算

**询问模板**：
- 对于 [组件 A]，最核心的功能是什么？可以独立交付吗？
- 这个任务是否可以进一步拆分？比如先实现基本功能，后续再补充扩展功能？
- 这个任务依赖哪些其他任务的产出？
- 你估算这个任务大概需要多少工作量？

**质量检查**：
- [ ] 任务是否覆盖了需求范围内的所有工作
- [ ] 每个任务是否有明确的验收条件
- [ ] 依赖关系是否形成 DAG（无环）
- [ ] 是否存在可以并行执行的任务
- [ ] 单个任务的粒度是否适中

**挑战时机**：
- 任务粒度过大 → 建议拆分为更小的可验收单元
- 任务之间依赖过于线性 → 讨论是否可以解耦以支持并行
- 工作量估算差异过大 → 建议重新评估或拆分

## 阶段 6：阶段规划与检查点

**阶段规划流程**：
1. 获取 deadline → 2. 初步划分阶段 → 3. 为每个阶段设计检查点 → 4. 将任务映射到阶段 → 5. 用任务工作量校验阶段时间 → 6. 如不合理则调整后回到步骤 2

**必须明确的要素**：Deadline、阶段划分、阶段检查点、任务-阶段映射、工作量校验

**询问模板**：
- 这个项目的 deadline 是什么时候？
- 你希望分几个阶段来交付？每个阶段的重点是什么？
- 每个阶段结束时，你期望看到什么样的可验证产出？
- 基于任务工作量估算，这个阶段安排的任务量是否合理？

**质量检查**：
- [ ] 每个阶段是否有独立的验证点（不依赖后续阶段即可验证）
- [ ] 阶段内的任务依赖是否在阶段内可闭合
- [ ] 工作量总和是否与阶段时间窗口匹配
- [ ] 高风险验证是否安排在早期阶段
- [ ] 关键路径是否已识别

**挑战时机**：
- 工作量超出阶段时间 → 讨论缩减范围或调整阶段
- 某阶段没有明确的检查点 → 要求定义可验证的产出
- 高风险任务放在了后期 → 建议提前以降低整体风险
- 关键路径过长 → 讨论是否可以通过调整依赖来缩短

## 阶段 7：风险评估与验证策略

### 7.1 技术风险识别

根据前面的讨论，主动评估：
- **选型风险** - 选择的技术/库是否存在不确定性？
- **性能风险** - 设计方案在预期负载下是否可行？
- **集成风险** - 组件之间、与外部系统之间的集成是否有不确定性？
- **规模风险** - 数据量/用户量增长后方案是否仍然适用？

### 7.2 验证策略

对每个高风险项制定验证计划：验证方式（spike/PoC/prototype）、验证目标、成功标准、失败预案、时间安排（通常应安排在早期）

**询问模板**：
- 在我们讨论的方案中，你觉得哪些部分的不确定性最高？
- 对于 [高风险点]，你觉得需要做什么验证来降低风险？
- 如果 [技术选型] 验证失败，你的 Plan B 是什么？

**质量检查**：
- [ ] 所有高不确定性的技术点是否已识别
- [ ] 每个高风险项是否有验证计划
- [ ] 验证是否安排在足够早的阶段
- [ ] 失败预案是否可行

**挑战时机**：
- 用户认为"没有技术风险" → 主动从选型、性能、集成等角度逐一审视
- 验证计划不够具体 → 要求明确验证目标和成功标准
- 高风险验证安排在后期 → 强烈建议提前，并说明延后的代价

**风险评估完成后**：检查是否需要将验证任务插入阶段规划，如需要则回到阶段 6 进行调整。

## 特殊场景处理

### 场景 1：需求文档不完整

1. 明确指出缺失的部分
2. 评估缺失信息对技术方案的影响
3. 建议先补充需求文档，或者在有限信息下标注假设继续推进

### 场景 2：技术选型分歧

1. 清晰陈述顾虑和理由
2. 请用户解释其选择的背景（可能有不了解的上下文）
3. 如果分歧无法解决，将两种方案都记录在决策文档中，标注分歧点

### 场景 3：需求变更

1. 记录需求变更点
2. 评估变更对已讨论的技术方案的影响
3. 建议用户先更新需求文档，再继续技术方案讨论
4. 如果变更较小，可以在当前讨论中处理并标注

### 场景 4：任务-阶段校验不通过

1. 明确展示不匹配的数据（时间 vs 工作量）
2. 讨论解决方案：缩减范围 / 调整阶段 / 增加资源 / 调整任务拆分
3. 调整后重新校验，直到通过

### 场景 5：中途修改

1. 接受修改请求
2. 评估修改对后续阶段的影响（波及分析）
3. 必要时重新讨论受影响的阶段
4. 更新决策记录（如果涉及选型变更）
