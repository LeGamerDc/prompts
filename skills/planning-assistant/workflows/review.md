# 评审工作流

## 单系统评审

载入：系统 memory + 系统全部文档 + vision

### 检查项

**1. 完整性**
- 是否有 `[TODO]` 或占位符未展开？
- 核心机制是否都有详细说明？
- 边界情况和异常处理是否覆盖？
- 数值框架是否完整？

**2. 一致性**
- 文档内部是否自洽（数值、术语、逻辑无矛盾）？
- 是否符合设计支柱？
- 术语是否与 `shared/glossary.md` 一致？

**3. 可行性**
- 设计是否过于复杂难以实现？
- 数值是否合理（无明显失衡）？
- 是否有过度设计的部分可以简化？

**4. 玩家体验**
- 该系统为玩家提供的体验是否清晰？
- 学习曲线是否合理？
- 是否有挫败感过高或无聊的设计？
- 玩家动机链是否完整（为什么玩家要参与这个系统）？

## 跨系统评审

载入：项目 memory + 所有相关系统 memory（不必载入全部文档）

### 检查项

1. 系统间接口是否双方都有定义且一致？
2. 共享约定（`shared/conventions.md`）是否被遵守？
3. 玩家体验是否连贯（系统切换时不会割裂）？
4. 是否有功能重叠或矛盾？
5. 资源/货币/进度等跨系统元素是否协调？

## 输出格式

```markdown
## 评审报告: <范围>
日期: YYYY-MM-DD

### 🔴 关键问题（必须解决）
- **问题**: 描述 | **位置**: 文件§章节 | **建议**: 解决方案

### 🟡 建议改进
- **问题**: 描述 | **位置**: 文件§章节 | **建议**: 改进方案

### 🟢 亮点
- 设计得好的部分及原因

### ❓ 待讨论
- 需要与用户讨论才能定论的设计决策
```

## 评审后操作

- 关键问题 → 写入对应 memory 的「待解决」
- 更新项目 memory 的「回顾」记录
- 如用户同意修改 → 切换到文档模式执行
