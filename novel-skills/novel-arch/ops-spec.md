# 运行态管理规范

> 触发条件：更新项目状态、记录决策、会话交接、任务管理。

`ops/` 管理项目的运行状态，确保跨会话的上下文连续性。它包含五类文件：

| 文件/目录 | 职责 |
|-----------|------|
| `status.md` | 项目当前状态快照 |
| `todo.md` | 分层任务清单 |
| `decisions.md` | 决策历史记录 |
| `handoffs/` | 会话交接记录 |
| `insights.md` | 创作经验收集池 |

同时，`index/unresolved-threads.md` 虽然存放在 `index/` 下，但其维护逻辑与 ops 密切相关，一并在此说明。

---

## 1. status.md 协议

### 1.1 用途

`ops/status.md` 是项目的**状态快照**，每次新会话开始时首先读取此文件以获取上下文。

### 1.2 必须包含的字段

```markdown
# 项目状态

## 当前阶段
[构思 | 大纲规划 | 第N卷写作 | 第N卷审稿 | 连载中]

## 进度概览
- **已完成卷数**：N
- **已完成章数**：N
- **总字数**：NNNNN
- **最新章节**：vol-XX/ch-XXX [章名]

## 已完成里程碑
- [YYYY-MM-DD] [里程碑描述]
- ...

## 下一步
- [ ] [最优先的行动项]
- [ ] [次优先的行动项]

## 阻塞项
[当前阻碍进展的问题，无则写"无"]

## 待确认决策
[需要人工确认的决策点，无则写"无"]
```

### 1.3 更新时机

| 事件 | 需要更新的字段 |
|------|---------------|
| 完成一章初稿 | 进度概览（章数、字数、最新章节）|
| 完成一卷 | 进度概览 + 已完成里程碑 + 下一步 |
| 做出重要决策 | 待确认决策（清除）|
| 发现阻塞问题 | 阻塞项 |
| 阶段切换（如从大纲进入写作）| 当前阶段 + 下一步 |

### 1.4 格式要求

- `last-updated` frontmatter 字段必须更新为当天日期
- "下一步"最多列 3-5 项，保持聚焦
- "阻塞项"如无阻塞，写"无"而非留空

## 2. todo.md 协议

### 2.1 用途

`ops/todo.md` 是项目的**任务池**，按类型和优先级组织。

### 2.2 任务分类

```markdown
# 任务清单

## 紧急
- [ ] [阻塞性任务，需要立即处理]

## 创作任务
- [ ] [写作相关：写章节、扩展大纲、创建角色卡等]

## 修订任务
- [ ] [修改相关：来自审稿的修正需求]

## 调研任务
- [ ] [研究相关：历史考据、桥段收集、竞品分析等]

## 设定任务
- [ ] [设定相关：补充世界观、完善力量体系等]
```

### 2.3 任务条目格式

```markdown
- [ ] [任务描述] `[优先级]` `[来源]`
```

- **优先级**：`P0`（阻塞）、`P1`（重要）、`P2`（普通）、`P3`（低优先）
- **来源**（可选）：`审稿`、`伏笔预警`、`用户指示`、`自动检查`

示例：

```markdown
- [ ] 补充陆父的角色卡 `P1` `用户指示`
- [ ] 修正 ch-005 中的地名错误 `P2` `审稿`
- [x] ~~完成 vol-01 卷纲~~ `P1`
```

### 2.4 与伏笔健康检查的联动

当伏笔健康检查发现预警时，自动在 todo.md 中添加对应任务：

```markdown
- [ ] 伏笔 F-003 已超过 8 章未推进，需补入推进事件 `P1` `伏笔预警`
```

### 2.5 维护规则

- 完成的任务标记 `[x]` 并加删除线
- 每卷完成时清理已完成任务，保持清单简洁
- "紧急"分类最多保留 3 个任务

## 3. decisions.md 协议

### 3.1 用途

`ops/decisions.md` 记录项目中做出的**所有重要决策**，方便回溯和审计。

### 3.2 条目格式

```markdown
### D-[NNN] [决策标题]
- **日期**：YYYY-MM-DD
- **决策**：[具体决策内容]
- **理由**：[为什么这么决定]
- **影响范围**：[影响了哪些文件/设定/情节]
- **状态**：confirmed | pending | revised
```

### 3.3 特殊决策类型

#### 大纲修订决策

当修改已有大纲时，需要额外记录：

```markdown
### D-015 修改第三卷主要矛盾
- **日期**：2026-03-15
- **决策**：将第三卷主要矛盾从"对抗边军"改为"粮价危机"
- **理由**：粮价危机更贴合种田文核心卖点，且与主角的经营能力直接关联
- **修改前**：对抗边军入侵，主角组织防御
- **修改后**：粮价被人为操控，主角以经营手段破局
- **连锁影响**：
  - vol-03 卷纲需重写
  - 角色"边军将领"角色定位需调整
  - 伏笔 F-008（边境暗流）回收时间推后
- **影响范围**：canon/outline/volumes/vol-03.md, canon/characters/supporting/border-general.md
- **状态**：confirmed
```

#### 设定变更决策

```markdown
### D-020 修改灵石兑换比例
- **日期**：2026-03-15
- **决策**：1 灵石 = 100 铜钱（原为 50 铜钱）
- **理由**：50 铜钱导致灵石价值感不足，无法支撑后续拍卖场景的戏剧张力
- **影响范围**：canon/world/economy/, 已发布 ch-001~ch-012 中涉及价格的描述
- **状态**：confirmed
```

### 3.4 分区管理

```markdown
## 已确认决策

[按时间倒序排列]

## 待确认决策

[需要人工拍板的决策，等待确认]

## 已废弃决策

[曾经确认但后来推翻的决策，保留备查]
```

## 4. 交接协议（handoffs/）

### 4.1 用途

`ops/handoffs/` 存放**会话交接记录**，确保不同会话（或不同 agent）之间的上下文连续性。

### 4.2 命名规则

```
YYYY-MM-DD-HHMM-[主题关键词].md
```

示例：`2026-03-15-1430-vol01-ch008-drafting.md`

### 4.3 Frontmatter

```yaml
---
type: handoff
date: YYYY-MM-DD
session-topic: [本次会话的主题]
---
```

### 4.4 必须包含的字段

```markdown
# 交接记录：[主题]

## 本次目标
[本次会话计划完成什么]

## 已完成
- [完成项1]
- [完成项2]

## 影响文件
- [被创建/修改的文件路径列表]

## 当前状态
[会话结束时项目处于什么状态]

## 下一步
- [ ] [后续会话应该做什么]
- [ ] [具体的行动项]

## 阻塞/待确认
[需要人工介入的事项，无则写"无"]
```

### 4.5 交接时机

| 场景 | 是否需要交接记录 |
|------|-----------------|
| 正常会话结束 | 是 |
| 完成重要里程碑（如完成一卷） | 是 |
| 做出重大决策后 | 是 |
| 阶段切换（如从大纲进入写作） | 是 |
| 简单的问答/小修改 | 否 |

### 4.6 与 status.md 的关系

- `status.md` 是**最新状态快照**，读取它即可了解项目全貌
- `handoffs/` 是**变更历史**，记录每次会话的具体工作内容
- 交接记录写完后，同步更新 `status.md` 中的相关字段

## 5. Unresolved Threads 维护

### 5.1 与伏笔矩阵的区别

| | 伏笔矩阵 | Unresolved Threads |
|---|---------|-------------------|
| **存放位置** | `canon/outline/foreshadowing.md` | `index/unresolved-threads.md` |
| **追踪对象** | 有意设计的叙事伏笔 | 所有类型的未闭合问题 |
| **类型** | 仅伏笔 | 伏笔未收、设定空洞、角色去向、逻辑缺口 |
| **管理者** | 创作时主动埋设和回收 | 审稿/检查时发现并记录 |

### 5.2 条目格式

```markdown
### T-[NNN] [线头描述]
- **来源**：[发现位置或时机]
- **类型**：伏笔未收 | 设定空洞 | 角色去向 | 逻辑缺口
- **优先级**：高 | 中 | 低
- **关联 canon**：[相关的 canon 文件]
- **说明**：[补充信息和处理建议]
```

### 5.3 更新时机

- 审稿发现问题时添加
- 一致性检查发现问题时添加
- 伏笔健康检查预警时添加
- 问题解决后移至"已解决"区，注明解决方式和日期

## 6. insights.md 协议

### 6.1 用途

`ops/insights.md` 是创作经验的**收集池**。agent 在任何创作工作中发现的经验、洞察、踩坑记录统一追加到此文件，在里程碑节点批量处理。

捕获行为的触发规则见 SKILL.md「全局约定 > 经验回流规则」——该规则始终生效，不依赖本文档是否被加载。

### 6.2 文件结构

```markdown
---
type: insights
last-updated: YYYY-MM-DD
last-processed: YYYY-MM-DD
---

# 创作经验收集池

## 待处理

[raw 状态的条目，按时间顺序追加]

## 已纳入

[accepted 状态的条目，记录去向]

## 不采纳

[rejected 状态的条目，记录理由]
```

### 6.3 条目格式

```markdown
### I-[NNN] [简短描述]
- **日期**：YYYY-MM-DD
- **来源**：[发现时的工作上下文]
- **类型**：best-practice | anti-pattern | conflict | observation
- **内容**：[经验描述]
- **状态**：raw | accepted | rejected
```

类型说明：

| 类型 | 含义 | 示例 |
|------|------|------|
| `best-practice` | 效果好的做法，值得复用 | "用天气描写做情绪过渡效果极佳" |
| `anti-pattern` | 导致问题的做法，应避免 | "单章信息灌输超过两个体系导致节奏崩溃" |
| `conflict` | 与 `core/` 现有规则矛盾 | "style-guide 要求不用第一人称，但回忆杀用第一人称更有代入感" |
| `observation` | 有趣的发现，暂不归类 | "读者对拍卖桥段的互动率明显高于日常" |

### 6.4 批量处理协议

**触发时机**：

- 每卷完成时
- 阶段切换时（如从大纲规划进入写作）
- `insights.md` 待处理条目累计超过 15 条时

**处理流程**：

1. **分类** — 逐条审视待处理 insights，按去向分组
2. **归纳** — 同类 insights 合并（如多条关于节奏的观察归纳为一条节奏原则）
3. **路由** — 确定每条 insight 应落地到哪个文件：

| insight 类型 | 典型去向 |
|-------------|---------|
| 写法/技巧类 | `core/style-guide.md` |
| 质量/标准类 | `core/quality-bar.md` |
| 流程/效率类 | `core/workflow.md` |
| 需修改设定 | `decisions.md` → `canon/` |

4. **冲突上报** — 类型为 `conflict` 的条目，或归纳后与现有规则矛盾的条目：
   - 在 `ops/status.md` 的「待确认决策」中列出
   - **不得自行修改 `core/` 中的任何规则**
   - 等待用户裁决后，按用户指示更新对应文件
5. **标记完成** — 处理完毕的条目移至「已纳入」或「不采纳」分区，注明：
   - `accepted`：去向文件路径 + 纳入日期
   - `rejected`：不采纳理由 + 日期

**处理后的条目示例**：

```markdown
### I-007 用天气描写做情绪过渡
- **日期**：2026-03-10
- **来源**：写 vol-01 ch-012 时
- **类型**：best-practice
- **内容**：在情绪转折的场景衔接处加入 1-2 句天气/环境变化描写，读起来自然不生硬
- **状态**：accepted
- **去向**：core/style-guide.md § 场景过渡技巧
- **处理日期**：2026-03-15
```

```markdown
### I-011 style-guide 要求不用第一人称，但回忆杀用更好
- **日期**：2026-03-12
- **来源**：写 vol-02 ch-003 主角回忆童年
- **类型**：conflict
- **内容**：当前 style-guide 规定全篇第三人称，但回忆段落用第一人称代入感更强
- **状态**：raw
- **冲突规则**：core/style-guide.md § 叙事视角
```

### 6.5 与其他 ops 文件的联动

| 事件 | 联动操作 |
|------|---------|
| 新增 `conflict` 类型 insight | 同步更新 `status.md` 的「待确认决策」 |
| 用户裁决 conflict | 结果写入 `decisions.md`，insight 标记 accepted/rejected |
| 批量处理完成 | 更新 `insights.md` frontmatter 的 `last-processed` 日期 |
